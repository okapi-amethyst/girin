# GIRIN プロジェクトルール

## ドキュメント構成

| ファイル | 役割 | AIが読む |
|---|---|---|
| `doc/PROJECT_PROMPT.md` | プロジェクト概要・フェーズ定義 | ✅ |
| `doc/02_企画設計/` | データ定義・仕様設計 | ✅ |
| `doc/03_詳細設計/` | コンポーネント・ページ実装仕様 | ✅ |
| `doc/CHANGELOG.md` | 変更履歴・解決済み経緯 | ❌ 人間のみ |
| `doc/実装未定.md` | 将来実装を検討する機能の備忘録 | ❌ 人間のみ |
| `doc/04_制作・実装/` | 進捗管理・環境設定・実装メモ | ❌ 人間のみ |

## ドキュメントフォルダ管理ルール

### フォルダ構成

**`01_リサーチ/` 〜 `04_制作・実装/` は Webサイト専用。**
Discord 関連は `doc/discord/`、派生企画は `doc/派生企画/` に置く。

```
doc/
  PROJECT_PROMPT.md       # プロジェクト全体
  CHARACTER_MASTER.md     # 共有マスターデータ
  CHANGELOG.md
  実装未定.md

  01_リサーチ/             # Webサイト専用
  02_企画設計/             # Webサイト専用
  03_詳細設計/             # Webサイト専用
  04_制作・実装/           # Webサイト専用

  discord/                # Discord Bot・サーバー設計
  派生企画/
```

### フォルダの判定基準

どのフォルダに置くか迷ったら、以下の問いで判定する。

| フォルダ | 問いかけ |
|---|---|
| `01_リサーチ/` | 「外の世界を調べた記録か？」（競合比較・用語集・ゲーム仕様調査） |
| `02_企画設計/` | 「何を・なぜ作るかの意思決定か？」（データスキーマ・記法仕様・方針） |
| `03_詳細設計/` | 「どう実装するかの仕様か？」（ページ設計・コンポーネント仕様・UI動作） |
| `04_制作・実装/` | 「作業の記録・状態管理か？」（進捗・環境設定・デプロイ手順） |

「リサーチ」という名称のファイルでも中身が実装仕様なら `03_詳細設計/` に置く。

### サブ仕様書の命名規則

1ページ・1機能の詳細設計が複数ファイルに分かれる場合は `ページ名_サブ機能.md` 形式にする。

```
03_詳細設計/
  コンボ投稿ページ.md          # 主設計書（フェーズ方針・フィールド一覧等）
  コンボ投稿ページ_入力UI.md   # サブ仕様（ボタンUI・トークンモデル等）
```

主設計書からサブ仕様書へリンクを張る。逆方向（サブ→主）もリンクを張る。

## Obsidian パス変換ルール

ユーザーが Obsidian のリンクコピー形式でパスを貼った場合、即座に以下の変換を行ってファイルを読む。

| Obsidian パス | プロジェクト内パス |
|---|---|
| `50_開発/GIRIN/` | `doc/` |

**変換例:**
- `50_開発/GIRIN/03_詳細設計/コンボカード・フィルタ` → `doc/03_詳細設計/コンボカード・フィルタ.md`
- `50_開発/GIRIN/02_企画設計/サイト構造設計` → `doc/02_企画設計/サイト構造設計.md`

拡張子が省略されている場合は `.md` を補完する。変換後のパスをユーザーに確認せず、直接そのパスで Read する。

## ドキュメント更新ルール

設計変更・仕様変更を行った際は、実装ファイルと **同じタイミング** で以下を更新すること。

### CHANGELOG への記録（`doc/CHANGELOG.md`）

以下のいずれかに該当する変更は `doc/CHANGELOG.md` に追記する。

**記録する:**
- データスキーマの変更（型・フィールド・選択肢の追加・削除・変更）
- UI仕様・フィルター動作の変更
- バグ修正（意図と挙動が一致していなかったもの）
- ファイルの追加・削除・移動
- 設計ドキュメントの大きな改訂

**記録しなくてよい:**
- スタイルの微調整（色・余白・フォントサイズ等）
- typo 修正
- コメント追加

**記述形式:**

```markdown
## YYYY-MM-DD

### カテゴリ名（例: データスキーマ / フィルターUI / バグ修正）
- 変更内容: 何を変えたか
- 理由: なぜ変えたか
```

### 設計書の課題・検討事項セクション

- **解決した項目** → 設計書から削除し、CHANGELOG に移動する
- **先送りにする項目** → `実装未定.md` に移動する
- 課題セクションには **未解決の問題だけ** を残す

## コード規約

- TypeScript Strict モード必須。`any` は使わない
- Astro コンポーネントのスタイルは `<style>` スコープ内に書く（グローバルに漏らさない）
- CSS カスタムプロパティは `global.css` で定義済みのトークンを使う。色・サイズのハードコードは避ける
  - 例外: バッジ色（`#60a5fa` / `#a78bfa` 等）はトークン化されていないため現状は許容
- マジックナンバーにはインラインコメントで理由を添える

---

## 実装ルール

- 既存ファイルを編集する前に必ず Read で現状を確認する
- 実装が設計書の仕様から外れていた場合は、実装完了後に設計書も更新する
- 設計書を更新した場合は「最終更新」日付も更新する
- 設計書（`doc/03_詳細設計/`）と実装が乖離していたら、どちらが正しいか確認してから修正する

---

## 禁止事項

- `doc/CHANGELOG.md` と `doc/実装未定.md` は実装の参照に使わない（設計書を参照する）
- 指示されていない機能を先回りして実装しない
- コミットは明示的に依頼された場合のみ行う
