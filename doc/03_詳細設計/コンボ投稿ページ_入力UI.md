# コンボ投稿ページ 入力UIサブ仕様

> 作成日: 2026-04-15
> 最終更新: 2026-04-17
> 目的: `/submit/` の入力表記フィールドにおける「ボタンクリックで記法テキストを生成する仕組み」の設計案
> 姉妹ドキュメント: `01_リサーチ/コンボ投稿ページリサーチ.md`（既存サイト比較）
> 設計への反映先: `コンボ投稿ページ.md`

---

## 1. 目的

管理者（自分）がコンボを手早く JSON に起こす際、入力表記を素早く・ミスなく書けるようにする。既存サイト（2XKOMBO）のビジュアルコンボビルダーを参考に、GIRIN の記法（`入力表記記法.md`）に合わせたボタンパッドUIを設計する。

**達成すべきこと:**

- ボタンクリックで `入力表記記法.md` に準拠したトークンを挿入できる
- テキスト直打ちも許可（コピペ・微修正のため）
- 長いコンボでも視認性が保たれる
- 入力ミスを巻き戻せる（UNDO / RESET）
- 記法の整合性を軽くチェックする（厳格にはしない）

## 2. UIレイアウト案

```
┌──────────────────────────────────────────────────────┐
│ 入力表記（テキストエリア・編集可）       [UNDO][RESET] │
├──────────────────────────────────────────────────────┤
│ ┌─────────────────────────────────────────────────┐  │
│ │  BD   AIR  D  │ Hold  dl.  HIT#  mash          │  │
│ │  7    8    9  │  L     M    H                  │  │
│ │  4    5    6  │  S1   S2    T                  │  │
│ │  1    2    3  │  >    ~    +    (½) (½)         │  │
│ └─────────────────────────────────────────────────┘  │
├──────────────────────────────────────────────────────┤
│ プレビュー（ComboCard風ライブ表示）                    │
└──────────────────────────────────────────────────────┘
```

**ボタン配置の根拠:**

| 行            | ボタン                                                  | 理由                                                                                     |
| ------------ | ---------------------------------------------------- | -------------------------------------------------------------------------------------- |
| 0行目（移動・修飾子行） | BD(44) / AIR(j.) / D(66) \| Hold / dl. / HIT# / mash | 左側に移動系（バックダッシュ・空中・ダッシュ）、右側に打鍵前に設定する修飾子。方向・攻撃ボタンの上に配置することで「修飾子 → 方向 → 攻撃 → 接続」の視線フローを実現 |
| 1行目          | 7 8 9 \| L M H                                       | 方向Numpad上段 + 攻撃上段（L/M/H）をコントローラー配列に準拠して左右に並べる                                          |
| 2行目          | 4 5 6 \| S1 S2 T                                     | 方向Numpad中段 + 攻撃下段（S1/S2/T）                                                             |
| 3行目          | 1 2 3 \| > ~ + ( )                                   | 方向Numpad下段 + 接続記号。`( )` は半幅で括弧ペアを視覚的に区別                                                |

**グリッド構造:** CSS Grid で左3列（方向Numpad）・右5列（攻撃・接続）に分割。0行目は左右を貫通する7列構成。攻撃ボタン（L/M/H・S1/S2/T）は接続ボタンより大きく（min-height 1.5倍程度）して主役感を出す。

**BD ボタン:** バックダッシュ。出現頻度は低いが方向系として0行目左側に配置。`44` の手入力でも代替可能。

**AIR（j.）ボタン:** 空中という場所条件を設定する修飾子。移動・位置系として BD / D と並べる。

**Hold ボタン（2XKOMBO準拠）:** クリックするとポップオーバーが出て `[ ]`（ホールド継続）/ `{ }`（任意タイミングで離す）を選択。

**HIT# ボタン:** クリックすると数字入力ポップオーバーが出て、直前ムーブに `(N)` を付与。

**6T / 2T / 4T について:** 方向+T の組み合わせで表現するため専用ボタンは設けない（方向パッドで `6` → ボタンパッドで `T` を押せばキャラ選択ポップオーバーが起動する）。

**レスポンシブ方針:** レスポンシブ対応あり（詳細は `コンボ投稿ページ.md` のレスポンシブ対応方針を参照）。ボタンパッドは一体のグリッドとして維持し、メタデータエリアがモバイルでは下段に落ちる構成を基本とする。

## 3. 入力の内部モデル

### 3-1. トークン列として管理

入力を単純な文字列ではなく **トークン列** として内部管理すると編集・UNDO が容易:

```ts
type Token =
  | { type: 'move'; text: string }                // '2M', 'j.H', '[6S1]', '66M'
  | { type: 'direction'; text: string }           // '6', '29' 等（単独方向トークン。2S1~6~2H の中間 6 など）
  | { type: 'connector'; text: '>' | '~' | '+' }
  | { type: 'bracket'; text: '(' | ')' }
  | { type: 'assist'; text: string }              // '6T:JINX' 等
  | { type: 'newline' }
```

レンダリング時にスペース区切りで結合 → `inputs` 文字列になる。

### 3-2. ムーブトークンの組み立て

`move` トークンは複数のパーツから組み立てる:

```
[方向プレフィックス] + [修飾子] + [攻撃ボタン or ホールドブロック] + [ヒット数指定]
```

例:
- `2M` = 方向 `2` + 攻撃 `M`
- `j.2H` = 修飾子 `j.` + 方向 `2` + 攻撃 `H`
- `dl.j.S2` = 修飾子 `dl.` + 修飾子 `j.` + 攻撃 `S2`
- `[6S1]` = ホールド `[` + 方向 `6` + 攻撃 `S1` + ホールド `]`
- `S2(2)` = 攻撃 `S2` + ヒット数 `(2)`
- `66M` = 方向 `6` + 方向 `6`（連続追記）+ 攻撃 `M`
- `29` = 方向 `2` + 方向 `9`（連続追記）→ コネクタで単独方向トークンとして確定

**バッファ方式（連続追記モード）:** 方向ボタンはバッファの「方向列」に**追記**される（上書きではない）。攻撃ボタンで確定するか、コネクタ（`>`/`~`）を押したとき方向だけ残っていれば単独方向トークンとして確定する。修飾子は順序保持で積まれる。確定前はテキストエリアにグレーでプレビュー表示。

## 4. ボタン動作仕様

### 4-1. 方向ボタン（1〜9）

- 組立バッファの「方向列」に**追記**する（上書きではない）
- `5` はニュートラル扱いで省略可
- 例: `6` → `6` → `M` で `66M`（ダッシュ攻撃）
- 例: `2` → `9` → コネクタ `>` で `29`（ハイジャンプ）が単独方向トークンとして確定し、次のムーブへ
- 例: `2` → `S1` → `~` → `6`（バッファに `6` 追記）→ `~` で `6` 単独確定 → `2` → `H` で `2S1 ~ 6 ~ 2H`

### 4-2. 攻撃ボタン（L / M / H / S1 / S2 / T）

- T を含む6ボタンが攻撃ボタン（公式コントローラー配列に準拠）
- バッファに方向・修飾子・ホールドフラグが溜まっていればそれを結合 → ムーブトークン確定
- 例: バッファ = `{dir:'2', mod:'j.'}` の状態で `H` → `j.2H` が追加されバッファクリア
- T を押した場合: バッファに方向が溜まっていれば `6T:JINX` 等のアシスト呼び出しとして処理する（キャラ選択ポップオーバーを起動）。方向なしの `T` は単独タッグ交代トークンとして追加

### 4-3. 修飾子ボタン（j. / dl. / mash.）

- `w.`（空振り）と `r.`（ダッシュ）は削除。ダッシュ攻撃は `66M` 等の連続方向入力で表現する
- バッファの「修飾子」スロットに積む（複数可、順序保持）
- 同じ修飾子を2回押したらトグル解除
- 例: `j.` → `dl.` → `H` で `dl.j.H`（内側から適用）

### 4-4. Hold ボタン（`[ ]` / `{ }` 選択ポップアップ）

- クリックするとポップアップが出て `[ ]`（ホールド継続）/ `{ }`（任意タイミングで離す）を選択
- 選択後、バッファに "ホールド待ち" フラグを立てる
- 次の攻撃ボタンで自動的に `[X]` or `{X}` に包んで確定
- 例: Hold → `[ ]` 選択 → `6` → `S1` → `[6S1]` が追加

### 4-5. 接続ボタン（`>` / `~` / `+`）

- connectorトークンを追加
- **方向列のみバッファに残っている場合**: 単独方向トークンとして確定してからコネクタを追加（`2S1 ~ 6 ~ 2H` の中間 `6` はこの挙動で生成される）
- 修飾子など方向以外がバッファに残っていたら警告（未確定ムーブがある）

### 4-6. 括弧ボタン（`(` / `)`）

- bracketトークンを追加（デュオ残キャラブロック）
- `(` と `)` の対応を追跡して不整合を警告

### 4-7. ダッシュボタン

- **D ボタン**: 方向列に `6` を2回追記した状態と同じ（次の攻撃ボタンで `66M` 等になる。攻撃なしでコネクタを押すと `66` 単独トークンとして確定）
- 方向パッドで `6` → `6` と押しても同じ結果になるが、1ボタンで入力できるショートカットとして残す

### 4-8. アシスト呼び出し（方向 + T）

- 方向パッドで `6` / `2` / `4` を押した後 `T` を押すとキャラ選択ポップオーバーを起動
- 選んだキャラのIDで `6T:JINX` のように挿入（大文字固定で表記ゆれ防止）
- キャラ選択肢は `CHARACTER_MASTER.md` の ID を利用
- 専用の `6T:▼` 等のボタンは設けない

### 4-9. HIT# ボタン（ヒット数指定）

- クリックすると数字入力ポップアップが出て、直前ムーブに `(N)` を付与
- 直前トークンが move でない場合は警告

### 4-10. UNDO / RESET

- **UNDO**: トークン列の末尾1つを削除（バッファが残っていればバッファのみクリア）
- **RESET**: トークン列・バッファを全クリア。確認ダイアログあり

## 5. バリデーション（Phase 2〜、軽め）

投稿前にチェック。エラー扱いではなく **警告表示** にとどめる:

| チェック項目 | 警告内容 |
|---|---|
| `(` と `)` の対応 | 「括弧が閉じていません」|
| 連続コネクタ | 「`>>` のように接続記号が連続しています」|
| 空のトークン列 | 「入力が空です」|
| `()` ブロックがあるのに `duo_only` 未チェック | 「デュオ残キャラブロックが含まれています。デュオ限定フラグを自動チェックしますか？」|
| ヒット数指定が move でないトークンの後 | 「ヒット数指定は move の直後にのみ使えます」|
| バッファに未確定ムーブが残ったまま送信 | 「未確定の入力があります」|

パーサは `入力表記記法.md` の仕様に準拠。フォーム側と表示側（ComboCard）で **共通のパーサモジュール** を使う想定。

---

## 6. グリフプレビュー（保留）

入力した記法をボタンアイコン（グリフ）で視覚化するプレビュー表示。

**保留理由:** グリフ表示はコンボ単体ページの設計で採用可否を決めてから設計する。閲覧側で使わない場合、入力UIのためだけにグリフレンダラーを実装するコストに見合わない。

**採用された場合の方針案:**
- PC: テキストエリアの上に常時表示
- モバイル: ボタンタップでポップアップ表示（ボタンパッドとの画面共存のため）

## 7. 実装メモ

- パーサ・レンダラは **フォーム側と表示側で共通化**。別実装すると記法の齟齬が生まれる
- 入力テキストは DB にはそのまま文字列で保存（`コンボ1枚フォーマット.md` の方針通り）
- ボタンUIコンポーネントは `src/components/ComboInputBuilder.astro`（仮）として単一コンポーネントに集約
- トークン管理ロジックは TypeScript で `src/lib/combo-input-tokens.ts`（仮）に分離 → ユニットテスト容易

### ペースト互換変換（2XKOMBO → GIRIN 記法）

テキストエリアへの `paste` イベント時に変換テーブルを通してから反映する。2XKOMBOからのコピペをそのまま受け入れるための互換レイヤー。

| 2XKOMBO記法     | GIRIN記法     | 備考                                  |
| ------------- | ----------- | ----------------------------------- |
| `D`           | `66`        | ダッシュ                                |
| `BD`          | `44`        | バックダッシュ                             |
| `r.`          | `66`        | ダッシュ攻撃                              |
| ` \| `        | （削除）        | 改行はしない                              |
| `MOVE(mash)`  | `mash.MOVE` | 連打                                  |
| `HOLD `       | `` （削除）     | HOLDキーワードのみ削除。`[X]`/`{X}` 括弧はそのまま保持 |
| `w.MOVE`      | `MOVE(0)`   | 0ヒットとして空振りとする                       |
| `T:CHARACTER` | `T`         | キャラクター表記はとる                         |

変換はペースト時のみ実行。テキスト直打ちには適用しない。
